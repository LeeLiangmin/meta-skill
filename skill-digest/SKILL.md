# Skill Digest

本 skill 的唯一目标：读取一个**已有的 skill**，产出一份**给人看的**理解简报，帮人在执行或信任这个 skill 之前，先建立一个准确的心智模型。

它的价值不在复述，而在**把隐含的显性化**——一个 skill 常常没明写"为什么这么做""什么算做完了""哪里最容易静默出错"，而这些恰恰是扫一眼 SKILL.md 最难看出、却最影响判断的东西。如果一个 skill 本身已经写得很透，这份简报最大的价值就只剩"缺口"一节，那也说明它没在做无用功。

## 三条不能越的边界

这三条决定了本 skill 是什么、不是什么：

- **描述，不评分。** 把影响判断的事实摆出来——假设、前置依赖、失败模式、模糊处、缺口——但不打分、不下"这 skill 好不好"的结论。要带证据的质量评判或执行审计，那是 skill-evaluator 的事。
- **缺口只列观察，不开处方。** 发现 skill 没说清的地方，如实记为缺口即可；不要替它写"应该怎么改"——那是 skill-creator 的事。
- **推断必须隔离标注，绝不冒充明说的。** 见下面"三种来源标注"。把推断混成事实，是本 skill 最严重的失败。

**本 skill 自身的类型：SOP/流程型 + 顾问/启发式型 混合** —— 五步流程是 SOP 骨架，三条边界和类型判定是顾问式原则。执行时按 SOP 型走流程，同时在简报中按需要补充顾问型小节（核心原则）。

（另外：本 skill 只做静态理解，不维护执行期的进度台账。）

## 流程（五步）

### 1. 定位并清点
`view` 目标 skill 的 SKILL.md。**关键：顺着它的引用往下读。** 现在的 skill 普遍用渐进式披露——SKILL.md 往往只是入口，真正的流程、验收标准、边界情况常常藏在它引用的 references/scripts/assets 里。只读 SKILL.md 抽出来的"流程"和"验收"几乎一定是残的。遍历它的文件树，但设一个合理深度、别无限展开；脚本看用途和接口即可，不必逐行通读。

### 2. 判类型
不同类型的 skill，该突出的信息完全不同。先判它属于哪一类（可能是混合）：

- **SOP / 流程型**——按步骤做、由验收门控（如走查、发布流程、合规检查）。重点是**验收标准**和**关键检查点**。
- **工具转换型**——给定输入产出特定输出（如 docx、pdf、数据清洗）。重点是**输入输出契约**和**边界情况**。
- **顾问 / 启发式型**——给原则和判断、没有固定流程（如设计指南、写作风格）。重点是**核心原则**和**适用 / 不适用的时机**。

判错类型会导致填出空洞或编造的小节（比如硬给顾问型 skill 填"验收清单"），所以这步先做。

判为混合型时：按主要类型（通常是 SOP 型）走输出结构，同时在核心原则 / 适用不适用等顾问型小节中补充判断原则。不要因为混合就省略任一侧的重要信息。

### 3. 按来源标注抽取
抽出的每一条信息，都要标明它的来源，三选一：

- **[明说]**——SKILL.md 或其引用文件里直接写着的。可附极短的定位（如指出在哪一节），但不要大段复制原文。
- **[推断]**——文件没明写，但由明说的内容合理推出的。**必须**让读者一眼看出这是你的推断、而非 skill 的原话。
- **[缺失]**——本该有、但 skill 没说清的（最常见于"为什么"和"完成定义"）。如实标缺失，不要替它脑补一个填上去。

这套标注是本 skill 可信的根基：读者据此判断哪些能直接信、哪些要自己再确认。

### 4. 按类型裁剪填表
用下面的输出结构，按第 2 步判定的类型决定哪几节是重点。超集里不适用的小节直接省略，不要硬填。

### 5. 输出前置式简报
人会跳读，所以最重要的信息放最前面（TL;DR），细节在后。落成一个 markdown 文件交付。

**交付后不要在对话中附加任何评判性评论或总结。** 简报本身就是全部交付物——额外评论（如"最大亮点""最脆弱环节"）破坏了"描述不评分"的边界。若确实有非评判性的发现想补充，应在简报的"缺口"节中呈现。

## 输出结构

把它当**超集**，按 skill 类型裁剪。全程带 [明说] / [推断] / [缺失] 标注。

```markdown
# <skill 名> 理解简报

## TL;DR
- **做什么**：一句话
- **何时用它**：触发的场景
- **用了得到什么**：产出
- **最大的风险点**：最该当心的一件事

## 类型
SOP / 工具转换 / 顾问启发式（或混合），一句话说明判断依据。

## 目标 / 完成定义
这个 skill 跑完、跑对之后，世界应该是什么样。尽量写成可检验的终态。

## 为什么
它为什么要这么做。标 [明说] 还是 [推断]；若 skill 通篇没交代，标 [缺失]。

## 前置条件 / 输入
开始前必须为真、或必须备齐的东西。

## 流程（阶段级）
按阶段、而非逐步。每个阶段写成三元组：
- **阶段**：这一阶段在干什么
  - 检查点：怎么确认这一阶段成了
  - 失败时：没成的话该怎么办 / 会怎样

（粒度停在"让人建立心智模型"即可，不下沉到复述每一步。）

## 关键观察点  ← 所有类型适用
如果只盯几个地方，盯哪几个——即最容易**静默失败**（出错了却不报错）的位置。

## 验收标准   ← SOP 型重点；其它类型可省
可逐条勾选、尽量客观的"算做完了"清单。

## 输入输出契约 + 边界情况   ← 工具型重点；其它类型可省
吃什么、吐什么、对什么格式 / 异常输入怎么处理。

## 核心原则 + 适用 / 不适用   ← 顾问型重点；其它类型可省
它给的判断原则，以及什么情况下不该套用。

## 失败模式 / 反目标  ← 所有类型适用
常见的踩坑、明确"不要做"的事。

## 缺口
skill 没说清的地方。**只观察，不开处方。**

---
- 想要带证据的质量评判 / 执行审计 → 用 **skill-evaluator**
- 想动手改进这个 skill → 用 **skill-creator**
```

默认将简报直接在对话中展示给用户；若用户明确要求落盘，则将简报写入 markdown 文件。

## 判类型的小例子

**例 1**——目标 skill 写着"按顺序执行下列步骤，每步完成后核对清单，全部通过方可发布"：这是 **SOP 型**，验收标准和检查点要重点抽。

**例 2**——目标 skill 写着"将用户上传的 .docx 转为带书签的 PDF"：这是 **工具转换型**，重点是输入输出契约（支持哪些 .docx 特性、遇到不支持的元素怎么办）。

**例 3**——目标 skill 写着"设计 UI 时优先考虑层次与留白、避免模板感"：这是 **顾问型**，没有固定流程，重点抽核心原则和不适用的场景；此时"验收标准"一节应**直接省略**，而不是硬编一个。
