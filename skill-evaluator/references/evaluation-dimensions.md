# 评估维度

下面七个维度覆盖了"一个 skill 好不好用"的主要方面。每个维度是一个**问题**，答案必须由 `evidence.json` 里的证据回答。不是每个维度都对每个 skill 都适用——不适用就在报告里标"不适用"并说明理由；缺少证据无法评的标"未评估"。

带 🔴 的维度需要**执行 session**才能评；没有 session 时这些维度标"未评估——缺少 session"。

---

## 1. 指令正确性（instruction_correctness）

**问题**：skill 给出的指令、命令、API 用法、事实陈述，本身是对的吗？

- 需要的证据：skill 文本（声称什么）+ external（独立事实，如命令实际输出、官方文档）。
- 通过：关键指令经外部核对属实，照做能成。
- 失败：skill 让你跑的命令不存在/参数错了、引用的 API 行为与文档不符、技术事实过时或错误。
- 常见失败：照搬了过时的库用法；命令在文档里是另一个名字；声称"会自动 X"但其实不会。
- 注意：正确性结论几乎都该走第 3 步校验，因为"我以为这命令对"恰恰是最容易出错的地方。

## 2. 内部一致性与完整性（consistency_completeness）

**问题**：skill 自相矛盾吗？有断头路或缺口吗？

- 需要的证据：skill 文本（含 references/scripts/assets 是否真实存在）。
- 通过：前后不打架；引用的文件都存在；完成声称的任务所需步骤齐全。
- 失败：A 处说先做 X、B 处说先做 Y；指向一个不存在的 reference；漏掉关键步骤导致照做会卡住。
- 常见失败：改了主文件忘了改 reference；scripts/ 里点名的脚本没附上；示例与正文规则冲突。

## 3. 可触发性（triggering）

**问题**：description 会在该触发时触发、不该触发时不触发吗？

- 需要的证据：description 文本；若有 session，还包括 skill 实际是否被 consult。
- 通过：描述既说清了"做什么"也说清了"何时用"，且能与相邻 skill（如易混的近义 skill）区分。
- 失败：描述太窄导致该用时不用（欠触发，这是常见病）；太泛导致抢了别的 skill 的活；与另一个 skill 描述高度重叠却没说清边界。
- 注意：严谨的触发率评估需要专门的 trigger eval（属于 skill-creator 的工具范畴）；本 skill 在没有那套数据时，只就描述文本和（若有）本次 session 的实际触发情况下结论，并说明这是有限证据。

## 4. 🔴 执行保真度 / 遵循度（execution_fidelity）

**问题**：session 里，模型实际按 skill 的指令走了吗？偏离在哪？偏离是帮了还是坏了事？

- 需要的证据：skill 文本（要求什么）+ session（实际做了什么），逐条对照。
- 通过：关键指令都被遵循；如有偏离，是合理的情境判断且没坏事。
- 失败：跳过了被强调的步骤（如"先读 SKILL.md"）；忽略了 bundled script 自己手写；按错的顺序做导致返工。
- 这是最依赖逐回合对照的维度：每条"遵循/偏离"结论都要同时引用 skill 侧和 session 侧的证据。

## 5. 🔴 产出质量（outcome_quality）

**问题**：最终产出达成了 skill 声称的目标 / 用户的请求 / 任何既定验收点了吗？

- 需要的证据：session 里的最终产出 + skill 声称的目标 + （若有）用户原始请求或 assertion。
- 通过：产出正确、落在正确位置、满足目标。
- 失败：产出有错、不完整、放错地方、或答非所问。
- 注意：质量判断容易主观。能客观核对的（文件是否存在、字段是否齐、数值是否对）就客观核对并引证据；确实需要主观判断的，标成 inferred 并写清判断依据，别伪装成硬事实。

## 6. 🔴 效率（efficiency）

**问题**：有没有无谓的步骤、重复劳动、上下文浪费？

- 需要的证据：session（回合数、工具调用、重试、绕路）。
- 通过：路径基本直接，没有明显被 skill 本身诱导出来的浪费。
- 失败：因为 skill 写得绕，模型反复试错；skill 让它读了一堆用不上的东西；多个步骤本可合并。
- 价值点：如果是 skill 的写法导致了浪费（而非任务本身复杂），这通常能直接转化成一条改进建议。

**量化检查清单**（评估时逐项核对）：
- [ ] **重复读取**：同一文件/资源被读取多次而无明显理由
- [ ] **冗余步骤**：存在可以合并的工具调用（如连续多次 write 到同一文件）
- [ ] **无效遍历**：读取了 skill 未要求且后续未使用的 references/scripts
- [ ] **重复劳动**：模型因 skill 指令模糊而反复尝试同一操作（如多次失败的编辑）
- [ ] **上下文膨胀**：session 中积累了大量与当前任务无关的历史上下文

**效率指标参考**（非绝对标准，仅作对比）：
- 回合数是否超过完成同类任务所需的最小回合数的 150%？
- 工具调用中 "读取" 类操作占比是否过高（如超过 60%）？
- 是否存在 3 次以上对同一目标的重复尝试？

## 7. 安全与意图一致性（safety_intent）

**问题**：skill 有没有做超出其自我描述的事？有没有危险指令？

- 需要的证据：skill 文本（+ 必要时 external）。
- 通过：行为与描述一致，无隐藏副作用，无危险/越权指令。
- 失败：描述说做 A 实际还偷偷做 B；包含会导致数据外泄、越权访问、破坏性操作的指令。
- 这条即便其它维度都好也要过一遍——意外性（surprise）本身就是缺陷。

---

## 落地提示

- 每个维度过完，把得到的结论写成 `findings.json` 里的 finding，`dimension` 字段用上面括号里的英文标识。
- 一个维度可以产出多条 finding，也可以一条都没有（说明这维度没问题——可在报告里用一句话带过，并引一两条支撑"无异常"的证据）。
- 高严重度的 finding（足以左右最终结论的）记得走第 3 步校验。
