# 校验协议

校验的目的，是用一个**独立视角**去检验那些一旦错了就会带偏整份报告的结论。评估者自己既读了证据又想下结论，天然容易自我印证；独立校验就是给这种确认偏误一个刹车。

## 何时必须校验

满足任一条，该结论就要走校验：

- **高严重度**：这条若成立，会显著改变最终结论（verdict）。`check_report.py` 会强制高严重度发现必须带校验记录。
- **来源冲突**：两个来源对同一事实说法不一致。
- **微妙解读**：结论依赖对 skill 文本的细读，容易看花眼（如"这句到底是要求还是建议"）。
- **外部事实**：结论依赖一个尚未落到 external 来源的事实（比如"这个命令其实不存在"——那就去把它验出来）。
- **自我怀疑**：你隐约觉得自己是在找证据支持已有结论，而非让证据说话。

低影响、来源明确、无歧义的 observed 事实，不必逐条校验——那样只是浪费。校验要用在刀刃上。

## 首选：subagent 独立校验

独立性是校验的全部价值。所以给 subagent 的信息要满足：**只给原始来源片段 + 一个精确的是非问题，绝不告诉它你的结论或你倾向的答案。** 否则它只会附和你。

**平台适配**：
- **OpenCode**：通过 `task` 工具派发校验任务给专用 subagent（`agents/evaluator-verifier.md`，install.sh 安装到 `.opencode/agents/`。只读、禁 bash）。child session 完成后其产出在约定路径读取。
- **Claude Code / MiMoCode**：使用 `actor` 工具 spawn 独立 subagent，只给原始材料和非问题。

subagent 提示词模板：

```
你在独立核实一项事实，请只依据我提供的原始材料作答，不要使用其它假设。

原始材料：
<把相关来源的原文片段贴在这里——可以是 SKILL.md 的某几行、session 的某几个回合、
 或某个命令的实际输出。只贴材料，不要贴任何评价。>

问题：<一个精确、可判定的是非问题。例：
 "在以上 session 片段中，模型在首次调用 write_file 之前，是否读取过该 skill 的 SKILL.md？">

请只回答：
- 结论：confirmed / refuted / inconclusive 之一
- 依据：指向上面材料的具体位置（行号或回合号）并简述
若材料不足以判定，答 inconclusive，不要猜。
```

把 subagent 的结论与依据记回 `evidence.json`（作为新的证据条目，`note` 写明来自校验），并在对应 finding 的 `verification` 字段里填 `method: "subagent"`、`result` 和它给出的 `evidence_ids`。

如需校验多条，可并行发多个 subagent，每个一问。

## 退化方案：无 subagent 时的"新视角复核"

有些环境（如 Claude.ai 网页端）没有 subagent。这时退化为新视角复核：

1. 把要核的结论先**搁置**，当它不存在。
2. 从零重新打开相关原始来源（重新 `view` 那几行、重读那几个回合），刻意不看自己之前的笔记和倾向。
3. 仅凭这次重读，独立回答那个精确的是非问题。
4. 把结果记回去，`method` 填 `"fresh_eyes"`。

务必在报告"方法学"一节如实说明：本次校验用的是新视角复核而非独立 subagent，其独立性较弱——因为复核者和评估者是同一个，确认偏误无法完全排除。诚实标注校验强度，本身也是"基于事实"的一部分。

## 校验结果三态

- **confirmed**：独立核实支持原结论。finding 可保留。
- **refuted**：独立核实推翻原结论。**改掉或删掉**那条 finding，别硬留。
- **inconclusive**：材料不足以判定。把该 finding 降级——要么降低严重度并注明"证据不足以确认"，要么移入"盲区与局限"作为假设。绝不把 inconclusive 当 confirmed 用。
